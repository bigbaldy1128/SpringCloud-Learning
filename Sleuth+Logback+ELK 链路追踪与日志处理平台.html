ELK

  1. ELK 是一套日志收集、存储、索引、分析与可视化展示的一体化平台,可以对分布式系统日志进行集中化管理和分析，帮助运维人员进行业务监控、故障定位及排除、业务趋势分析、安全与合规审计：

    • 其中 LogStash 负责不同来源的日志的收集与预处理（可以通过安装不同的插件，从数据库如Redis、文件、消息队列如Kafka、TCP/UDP连接等 input 读取日志，通过自定义 filter 解析日志，格式化输出为 json/plaintext等格式便于后续应用）；
    • Elasticsearch 则负责为上游LogStash处理后的格式化日志建立索引，提供准实时的查询服务，其中索引字段等信息由Logstash output配置；
    • 而 Kibana 负责从ES中检索数据，（自动）分析数据间的关联，实提供各种查询统计和Web界面可视化展示功能。

  3. Logstash本身就像一个 数据输入-编码处理解码-输出 的多线程管道，数据在线程之间以事件的形式流动，Logstash 会给事件添加一些额外信息比如最重要的 @timestamp，用来标记事件的发生时间。其他的比如：host （通常为IP）、type （事件类型）、tags （事件属性数组）；

    • 通常根据不同任务内容将Logstash组件（线程）分为 Shipper 负责收集日志、Indexer 负责日志存储、Broker 可以看作是日志集线器/缓冲器，可以连接多个 Shipper 和多个 Indexer（通常配置一个Redis或Kafka作为Broker，不过也可不让Logstash存储，数据直接对接ES）。
    • 也可从数据角度看，Input插件读取多源数据，Filter插件负责解析处理数据，Output插件负责输出数据，Filter插件具有比较完备的功能，比如grok，能通过正则解析和结构化任何文本，但是也有实验提到grok功能强大但效率不高，可以将数据处理交给后端的Index，而Filter只做多行匹配。

  4. Elasticsearch是一个基于Lucene的开源搜索引擎（倒排索引，根据关键词检索文档ID），其相关介绍可以参考ES中译文档。在使用ES前需要先了解几个概念：

    • 以传统数据库类比，一个Index相当于一个数据库，一个Document Type即其中的一张表，一条Document即某一行一条记录，每一条Document可以分为多个Field，即表中的某一列。或者以列式数据库（如HBase）类比，一个Index相当于一张表，一个Docment Type相当于一个Column Family，而每一个Field对应一个Column；
    • 创建Index前，必须根据数据规模、集群机器数量规定好该Index的分片数量即Shards，一般默认分片数为5，则数据哈希值%5决定被该条数据存储到哪个Shard中，而Shard则均匀的分布在集群各机器中；另外为了防止数据丢失、实现负载均衡，ES默认数据冗余度为2，即还会有5个Replica Shards，且备份Shard会被尽量分配到不同的机器上，以防止某台机器故障导致数据访问完全失效。
    • 一个ES Cluster 由多个ES服务实例组成，运行在JVM环境中，且对内存需求较高。为了集群保证服务效率和健壮性，一般一个主机只运行一个服务实例（也可以在单台服务器开多端口、起多实例上实现伪集群），各个实例即ES Node间借助zookeeper实现高可用（如选举、发现、移除），集群中每一个Client Node（Node分为Master Node/Data Node/Client Node(Ingest Node，Coordinating Only Node)）都知道某份文档保存在哪一个Shard中，通过请求转发、合并结果的方式实现负载均衡。
    • ES默认接收Json作为格式化输入，并且默认保存源文件（可以当Nosql用），每一个ES Field为key-value形式数据，创建Index时指定每个Field value的数据格式（如String/Integer/Date）、是否需要索引、是否需要分词、分词用Analyzer、搜索用Analyzer。ES(lucene)通过Analyzer实现对字段的分词，其处理流程分为三步：①CharFilter 字符过滤比如大小写替换 ②Tokenizer 分词器，比如英文按空格分词 ③TokenizerFilter 对上一步得到的Token进一步处理，比如添加同义词等，形成Term集合。通过配置Index Mapping实现自定义Tokenizer、自定义Analyzer。
{"index": {
     "analysis": {
         "analyzer": {
		<!--自定义id_analyzer,分词器为id_tokenizer-->
             "id_analyzer": {
                 "type": "custom",
                 "tokenizer": "id_tokenizer"
             },
             "index": {
                 "type": "index_ansj",
             },
             "query": {
                 "type": "query_ansj",
             }
         },

         "tokenizer": {
             "id_tokenizer": {
                 "pattern": "-",
                 "type": "pattern"
             }
         }
     }
 }
  1. ELK平台中，Logstash实现了很多对ElasticSearch的自动配置，但是并没有进行完全的优化，在处理数据入 ES 时以应用为单位创建索引，索引按照天来拆分，这样更符合查询的业务需求，但是也增加了Index的数目，一般定期将日志数据导入hdfs，ES中日志数据保留数月。

Logback

  1. Logback是由log4j创始人设计的另一个开源日志，其功能更多、效率更高、稳定性更强。Logback的核心对象包括：Logger、Appender、Layout。其中Logger是日志的记录器，需将它关联到对应的context，用于存放日志对象，也可以定义日志类型、级别；Appender用于指定日志输出的目的地，可以是控制台、文件、远程套接字服务器或服务器等； Layout负责把事件转换成字符串，格式化的日志信息的输出。Appender、Layout是最消耗性能的操作。
  2. Logback组件是每一个application内部日志收集器，需要通过 logstash-logback-encoder 将logstash与logback连接起来；logback通过定义各种encoder来实现输出到logstash的日志格式，其配置可参考上面的链接，如下示例最终生成一个json:{[log_lever],[serviceName],[traceID],[threadName],[msg],[id]}：
	<!--自定义springAppName字段-->
   <springProperty scope="context" name="springAppName" source="spring.application.name"/>
   
   <!--创建名为LOGSTASH的appender，设置TCP服务端口、日志级别、encoder-->
   <appender name="LOGSTASH" class="net.logstash.logback.appender.LogstashTcpSocketAppender">
   
       <destination>172.24.62.181:4560</destination>
   	
       <!--<filter class="ch.qos.logback.classic.filter.ThresholdFilter">-->
           <!--<level>INFO</level>-->
       <!--</filter>-->
   	
       <encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
           <providers>
               <pattern>
                   <pattern>
   				<!--定义输出样式，这里以json格式输出提高logstash匹配解析效率，其中涉及到layouts函数及logback mdc概念，参考后文介绍-->
                       {"log_level": "%level","serviceName":"${springAppName:-}","traceID":"%X{X-B3-SpanId:-}","threadName":"%thread","msg":"%message","id":"%X{id:-}"}
                   </pattern>
               </pattern>
           </providers>
       </encoder>
   </appender>

   <include resource="org/springframework/boot/logging/logback/base.xml"/>
   <!--设定默认log配置为根logger配置-->
   <root level="INFO">
       <appender-ref ref="LOGSTASH" />
   </root>
   <!--指定codesafe-system包及其子包所有类使用以下logger配置，additivity表示不继承父Logger配置，所有logger配置继承自根logger-->
   <logger name="com.qihoo.codesafe.system" level="info" additivity="false">  
   	<appender-ref ref="CONSOLE" />
   </logger>
  1. 注意到上面的示例中，pattern字段包含很多取值表达式如%level、%X{id:-}，其中前者表示从logback日志中取出level字段，后者表示从logback mdc内容中取出id段。其表达式相关信息见官方文档 Logback_Layouts、源码 PatternLayout；常见如%c{length}(即%logger{length})限制输出长度、%d{pattern}(即%date{length})限制日期格式、%m(即%message)输出应用程序给输出到logger的内容字段、%X{key:-defaultVal}(即%mdc{key})根据key输出MDC字段...
  2. 上文提到logback的MDC，即Mapped Diagnostic Context，其详细介绍见官方文档Logback_MDC，MDC使得logback在分布式、多线程系统中，能对线程产生的日志实现动态标记，方便定位。MDC实际定义了一个InheritableThreadLocal Map，保证线程间安全同时使得子线程同样具备父线程变量视图（摘自官方文档中文翻译；例如通过MDC.put("id",myTheadId)那么该线程内的所有logger.info()/logger.warn()...都会带有 id 键值对，无需每一条记录中手动添加该 id 信息，取出时不通过%m取出message字段内容再解析，而通过%X{id}即可直接取出。
  3. SLF4J(Simple Logging Facade for Java)只是一个日志收集门面，只是一系列的日志接口，而log4j/logback是具体实现了的日志框架(logback性能更高)；这样做的好处是，不同应用（类库）可以使用不同的日志框架，但是应用调用统一的抽象层slf4j接口，实现解耦；同样MDC也是slf4j中提出的，在logback中具体实现了其。下面的代码中并没有出现Logback相关的代码，是slf4j通过StaticLoggerBinder绑定了Classpath下的Logback。
  import org.slf4j.Logger;
  import org.slf4j.LoggerFactory;
  private static final Logger logger = LoggerFactory.getLogger(SimpleDemo.class);

Sleuth

  1. Zipkin：调研过程中发现Sleuth和Zipkin关系密切，Zipkin是Twitter开源的分布式链路调用监控系统，而Sleuth的实现是在Zipkin的基础上进行了封装（Sleuth通过在微服务日志中加id、聚合日志的方式分析调用情况，而Zipkin服务可以可视化显示调用链路、请求故障），Zipkin可以用过9411端口访问，如下图，是四个微服务组件间的依赖调用关系：
  2. 参考官方文档，需要明白以下几个概念，尤其Span及Trace

Span：基本工作单元，例如，在一个新建的Span中发送一个RPC等同于发送一个回应请求给RPC，Span通过一个64位ID唯一标识，trace以另一个64位ID表示，span还有其他数据信息，比如摘要、时间戳事件、关键值注释(tags)、Span的ID、以及进度ID(通常是IP地址)，Span在不断的启动和停止，同时记录了时间信息，当你创建了一个Span，你必须在未来的某个时刻停止它。
Trace：一系列Spans组成的一个树状结构，例如，如果你正在跑一个分布式大数据工程，你可能需要创建一个Trace。
Annotation：用来及时记录一个事件的存在，一些核心annotations用来定义一个请求的开始和结束。

3. Trace可以理解成一次完整的微服务调用链，而链上的每一个节点，即各个微服务组件间的通信（请求或回应），每一个Trace有一个ID，并且可以自定义一些tag方便追踪，而所有Span都会记录1SpanID、父一级Span的SpanID(16进制的长整数字符串)、所在Trace的TraceID，是否导入Zipkin的Exportable标记，这样便可以实现链路的追踪和溯源。
同样可以通过给Span附加信息来实现对Span的定制，主要有两种手段：addBaggage和addTag，其对比在官方文档中介绍比较清楚：baggage信息会随着父Span传递至所有子Span，而tag信息只是给某个Span单独带上标记。
4. Sleuth通过注入MDC的方式向Logback的每一条日志加上追踪信息，相关源码

		MDC.put(Span.SPAN_ID_NAME, Span.idToHex(span.getSpanId()));
		MDC.put(Span.SPAN_EXPORT_NAME, String.valueOf(span.isExportable()));
		MDC.put(Span.TRACE_ID_NAME, span.traceIdString());
		if (parent != null) {
			MDC.put(Span.PARENT_ID_NAME, Span.idToHex(parent.getSpanId()));
		}

可以看出sleuth将SpanID、TraceID、Parent_Span_ID、Exportable信息注入MDC，与应用中用户提交的log event信息融合，最终形成完整的log信息。
